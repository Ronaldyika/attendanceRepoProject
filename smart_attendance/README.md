## Overview

This backend provides the complete server-side API for a fraud-resistant, offline-capable academic attendance system. The key technical innovations are:

| Feature | Mechanism |
|---|---|
| Offline operation | Flutter client + SQLite; server never required at scan time |
| Fraud prevention | HMAC-SHA256 signed QR codes + device UUID binding |
| Duplicate safety | Idempotency keys + `UNIQUE(student, session, device_uuid)` |
| Sync reliability | Idempotent batch endpoint; conflict & integrity audit logs |
| Security | JWT auth, token blacklisting, role-based permissions |

## Quick Start

### 1. Clone and set up the environment

```bash
git clone <your-repo-url>
cd smart_attendance
python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env
# Edit .env with your values
```

### 2. Database setup

**Development (SQLite – default):** no extra configuration needed.

**Production (PostgreSQL):**
```sql
CREATE DATABASE smart_attendance_db;
CREATE USER attendance_user WITH PASSWORD 'your-password';
GRANT ALL PRIVILEGES ON DATABASE smart_attendance_db TO attendance_user;
```
Then set `DB_*` variables in `.env`.

### 3. Run migrations and create superuser

```bash
python manage.py migrate
python manage.py createsuperuser
```

### 4. Start the development server

```bash
python manage.py runserver
```

### 5. Run tests

```bash
python manage.py test --verbosity=2
# Expected: 24 tests, 0 errors, 0 failures
```

#### POST `/auth/login/`
Obtain JWT tokens. Supply `device_uuid` to trigger device binding.

**Request:**
```json
{
  "email": "student@nahpi.cm",
  "password": "SecurePass123",
  "device_uuid": "android-device-uuid-from-device_info_plus"
}
```

**Response 200:**
```json
{
  "access": "<JWT access token>",
  "refresh": "<JWT refresh token>",
  "user": {
    "id": "...", "email": "...", "full_name": "Ronald Buhnyuy",
    "role": "student", "device_uuid": "android-device-uuid-...",
    "device_bound_at": "2025-04-01T08:00:00Z"
  }
}
```

> **Fraud prevention:** If a different `device_uuid` is submitted for an already-bound account, the login is **rejected with HTTP 400**.

### ATTENDANCE Endpoints  (`/api/v1/`)

#### POST `/attendance/scan/`  *(Student only)*
Record attendance via a real-time online scan.

**Request:**
```json
{
  "session_id": "<session-uuid>",
  "qr_payload": "<full qr string from scan>",
  "device_uuid": "<student device uuid>",
  "scanned_at": "2025-04-01T08:05:00Z"
}
```

**Server validates:**
1. Student role.
2. Device UUID matches registered device.
3. Session is open and not expired.
4. HMAC-SHA256 signature is valid.
5. QR has not expired (with clock-skew tolerance).
6. No duplicate record for this session.

**Response 201:** `{ "message": "Attendance recorded.", "record": { ... } }`

**Rejection reasons (HTTP 400):**
- `device_uuid`: Device not bound to this account.
- `qr_payload: QR verification failed: invalid_hmac` — tampered/forged QR.
- `qr_payload: QR verification failed: expired` — screenshot attack.
- Duplicate record.

---

#### GET `/attendance/records/`
List attendance records. Supports `?course_id=<uuid>&session_id=<uuid>`.

### SYNC Endpoints  (`/api/v1/`)

#### POST `/sync/`  *(Student only)*
Submit a batch of offline-captured attendance records.

**Request:**
```json
{
  "device_uuid": "android-device-uuid",
  "records": [
    {
      "session_id": "<uuid>",
      "device_uuid": "android-device-uuid",
      "scanned_at": "2025-04-01T08:05:00Z",
      "idempotency_key": "device-uuid|session-uuid|timestamp-ms",
      "hmac_signature": "<hex>",
      "qr_payload": "<full qr string>"
    }
  ]
}
```

**Server-side validation pipeline (per record):**
1. Session resolution.
2. Device UUID match.
3. Cross-device conflict detection.
4. HMAC-SHA256 re-verification.
5. Clock-skew check.
6. Idempotency key duplicate check.
7. INSERT (idempotent).

**Response 200:**
```json
{
  "batch_id": "<uuid>",
  "total_submitted": 5,
  "accepted": 4,
  "rejected": 0,
  "duplicates": 1,
  "conflicts_flagged": 0,
  "status": "complete"
}
```

#### GET `/sync/history/`
List all sync batches for the authenticated user.

## QR Payload Format

```
<session_id>|<course_code>|<expiry_unix>|<hmac_sha256_hex>
```

**Example:**
```
3f2a1b4c-...|CPE501|1743494100|a3f9b2c1d4e5f6...
```

**HMAC computation:**
```python
import hmac, hashlib
payload_base = f"{session_id}|{course_code}|{expiry_unix}"
signature = hmac.new(
    secret.encode(), payload_base.encode(), hashlib.sha256
).hexdigest()
```

The `session_secret` is generated server-side using `secrets.token_hex(64)` and **never transmitted to student devices**.

## Security Architecture Summary

```
┌─────────────┐          ┌──────────────────────────┐
│  Lecturer   │          │  Student Device          │
│  Device     │          │                          │
│             │──────────│  1. Scans QR             │
│  Displays   │  QR Code │  2. Verifies HMAC locally│
│  QR Code    │          │  3. Checks expiry        │
│             │          │  4. Checks device_uuid   │
└─────────────┘          │  5. Checks local dupes   │
                         │  6. Writes to SQLite      │
                         └────────────┬─────────────┘
                                      │ (when online)
                         ┌────────────▼─────────────┐
                         │  Django REST Backend      │
                         │                          │
                         │  7.  Re-verify HMAC      │
                         │  8.  Re-check device     │
                         │  9.  Clock-skew check    │
                         │  10. Idempotency check   │
                         │  11. Conflict detection  │
                         │  12. INSERT PostgreSQL   │
                         └──────────────────────────┘
```
